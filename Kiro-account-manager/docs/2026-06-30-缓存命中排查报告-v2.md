# Prompt Cache 命中率排查报告 v2（2026-06-30）

## 背景与症状

线上（10.0.1.116）升级后，抓包「缓存命中分析」显示命中率 **0%**：所有会话
`cacheRead=0`，断裂点统一报 `system_block_changed system[0]`。真实多轮 Claude Code
会话也持续 0% 命中，每个请求都按全量 input（~18 万 token）计费，成本是本可省下的数倍。

时间窗：约 3 天前的抓包仍有命中（单会话 cacheRead 43 万 / 97 万），之后归零。
期间变化：① Claude Code CLI 2.1.193 → 2.1.195（`cache_control` 新增 `scope:global`、
system 由 3 块变 4 块）；② 反代 app 升级（但缓存相关代码自 1.7.11 未改）。

## 排查方法

1. **抓包结构分析**：解析 `~/Library/Application Support/kiro-account-manager/captures/`
   下的 `*.meta.json`（含 Kiro 解析后 usage）与 `*.json`（入站原始请求体）。
2. **116 实时回放**：用抓到的真实 body 朝 `localhost:5580` 连发，控制会话粘性
   （`x-session-affinity` 头）钉定账号，观察响应里的 `cache_read/creation_input_tokens`。
3. **本地真实转换**：用 esbuild 把 `translator.ts` 打成独立 bundle（桩掉 electron 依赖），
   离线跑真实 `claudeToKiro`+`buildKiroPayload`，逐字节 diff 连续两轮转换后的 Kiro payload。
4. **本地调试 dump**（`KIRO_CACHE_DEBUG=1`，已在收口时移除）：落盘发往 Kiro 的完整
   payload，定位真实发送字节的分叉点。

## 关键发现

### 发现 1：抓包分析器误报（已修）
`captureAnalyzer.systemBlocks` 在 diff 前**未剥离** `x-anthropic-billing-header` 行，
而该行（`cch` 每请求变化）在**转发 Kiro 前**已被 `translator.stripBillingHeader` 剥掉。
于是分析器把「转发前就会被剥、不影响 Kiro 缓存」的 billing 行误判成 system[0] 断裂点。
- **修复**：`systemBlocks` 取文本时同样 `stripBillingHeader`，与转发链路一致。
- **验证**：修复后断裂点原因从 `system_block_changed` 变为 `unknown`（内容前缀确实未变）。

### 发现 2：时间戳毒化 prompt cache（已修）
`translator.prependTimeContext` 给每个请求的**当前消息**注入
`[Context: Current time is <ISO>]`。问题链：
- 当前消息带 cachePoint，会被 Bedrock 缓存；
- 这一轮的当前消息，**下一轮原样进入 history**，但 history 版本**不含**该时间戳前缀；
- 于是缓存段末尾每轮错位，前缀永远对不上。

字节级实证：去掉时间戳前缀后，本轮当前消息与下一轮 history 中的同条消息**逐字相同**
（`req-0002` 当前消息 == `req-0004` history[198]）。
- **修复**：`prependTimeContext` 改为内容透传，不再注入时间戳。当前时间由客户端
  自身环境上下文提供。

### 发现 3：边界 cachePoint 方案无效（已回退）
曾尝试给「最近一条已完成的 user 轮次」补 cachePoint，制造跨轮复现的缓存边界。
真实发送字节对比证明它**反而添乱**：cachePoint 作为消息上的字段，每轮落在不同消息上
（B→history[196]，C→history[198]），导致同一条消息在相邻两轮的序列化字节不同
（一轮带 `cachePoint` 字段、一轮不带），缓存前缀正好断在该处。已回退。

### 发现 4：根本机制 —— Kiro 仅认末条 cachePoint（核心遗留）
对照实验（同账号、会话粘性钉定、同模型 opus-4.8、CodeWhisperer 端点）：

| 场景 | 结果 |
|---|---|
| A：req-0002 冷启动 | `cache_creation=104782`（写） |
| B：req-0002 **完全相同**再发 | `cache_read=104782`（**命中**） |
| C：req-0004 **增长一轮** | `cache_creation=105061`、`cache_read=0`（**不命中**） |

- B 能命中 → 账号/端点/基础设施均正常，**完全相同的请求可命中**。
- C 不命中，尽管：转换后 B 的 history **100% 是 C 的干净字节前缀**、conversationId 稳定、
  `system(history[0])` 的 cachePoint 也存在。
- 推断：**Kiro/Bedrock（`generateAssistantResponse`）只在"最后一条消息"的 cachePoint
  位置匹配缓存，且要求 `[0..末条cachePoint]` 整段与某次历史写入完全一致才读**；
  `history` 中（含 system[0]）的 cachePoint 不产生读。增长对话里末条 cachePoint 每轮前移、
  且上一轮当前消息进 history 后丢失 cachePoint 字段 → 整段永远对不齐 → 0 命中。

### 未能完全复现的点（诚实记录）
回放老格式（2.1.193）的 `e3f034ad` 会话，`seq14`（增长、共享前缀）实测读到
`cache_read=314488`——即**老格式在某次测试中实现了增量前缀命中**。但同样手法在新格式
（2.1.195）上不灵。两者转换后 payload 结构一致（cachePoint 位置、前缀关系、模型相同），
差异仅在两段对话的**具体内容字节**。该差异落在 Bedrock 内部匹配行为，**代理外部无法观测**，
未能定论新旧差异的确切原因。

- ✅ 修复抓包分析器误报（`captureAnalyzer.ts`）—— 断裂点如实显示。
- ↩️ 回退边界 cachePoint 方案；移除调试 dump（仅本地排查用）。

  A 写 / B 命中 / 无调试残留 / `/admin/stats` 200。

## 遗留与后续建议

**增长型多轮对话的缓存命中，代理侧目前无法根治**，根因是 Kiro 暴露 Bedrock 缓存的方式
（疑似仅认末条 cachePoint、不做更短前缀的跨轮匹配）。后续方向：

1. **走 Kiro / AWS 侧确认**：`generateAssistantResponse` 是否支持 history 中间消息的
   cachePoint、是否支持"末条 cachePoint 之前的更短前缀"自动命中。这是能否根治的关键。
2. **观察 Claude Code 格式演进**：2.1.193 在测试中可增量命中、2.1.195 不行，若后续版本
   调整 `cache_control`（如去掉 `scope:global`）可能自然恢复。
3. **如需在代理侧再尝试**：方向是让被缓存前缀（含 cachePoint 标记）跨轮**字节级稳定**，
   例如将 cachePoint 固定在不随对话增长漂移的位置——但受发现 4 的限制，可行性需先由第 1 点确认。
